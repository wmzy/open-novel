# open-novel 工具使用指南

> open-novel 是 AI 驱动的长篇小说创作系统。
> 技术栈：React 19 + Hono + PGlite + Drizzle + Linaria + Vite + AI Agent（Claude Code / OpenCode / Oh My Pi）

---

## 1. 快速开始

### 1.1 启动开发服务器

```bash
pnpm install
pnpm dev          # 启动 vite dev server（含 API），默认端口 3006
```

生产构建：

```bash
pnpm build        # build:client + build:server
pnpm start        # node dist/server/api.js
```

### 1.2 前置条件

- 至少一种 AI Agent CLI 已安装并在 PATH 中：
  - **Claude Code**：`claude --version`
  - **OpenCode**：`opencode --version`
  - **Oh My Pi (omp)**：`omp --version`（需支持 `omp acp` 子命令）
- agent 超时配置：环境变量 `AGENT_TIMEOUT_MS`（默认 1800000 = 30 分钟）

## 2. 创作流程（六阶段）

open-novel 的创作分六个阶段，每个阶段由 agent 通过 API 触发，产出写入 `.novel/` 目录：

```
concept → world → characters → outline → scenes → writing
 构思     世界观    角色档案     大纲      场景表     正文写作
```

### 阶段产出文件

| 阶段 | 产出文件 | 内容 |
|---|---|---|
| concept | `.novel/concept.md` | 核心概念、前提、主要冲突 |
| world | `.novel/world-building.md` | 世界设定、规则、历史、文化 |
| characters | `.novel/characters/profiles.md` | 角色动机、背景、关系、弧光 |
| outline | `.novel/outline-detailed.md` | 逐章大纲（幕、节拍、字数分配） |
| scenes | `.novel/scenes.md` | 逐章场景（主动 Scene / 被动 Sequel） |
| writing | `.novel/chapters/第N章.md` | 正文 + `.summary.md` + `state.json` |

### 阶段推进

每阶段完成后，agent 通过 PATCH 推进阶段：

```bash
curl -X PATCH http://localhost:3006/api/projects/{projectId} \
  -H "Content-Type: application/json" \
  -d '{"currentStage": "world"}'
```

## 3. 核心 API

### 3.1 项目管理

```bash
# 创建项目
POST /api/projects
{
  "title": "令牌",
  "genre": "wuxia",
  "chapterCount": 20,
  "targetWords": 80000
}

# 获取项目信息
GET /api/projects/{id}

# 更新阶段
PATCH /api/projects/{id}
{ "currentStage": "writing" }

# 生成逐章模板脚手架（大纲/场景）
POST /api/projects/{id}/generate-templates

# 预览模板（不落盘）
GET /api/projects/{id}/templates/outline-detailed
GET /api/projects/{id}/templates/scenes
```

### 3.2 触发 Agent 写作（核心）

```bash
POST /api/runs
{
  "projectId": "proj_xxx",
  "agentId": "claude",        # claude | opencode | omp（前端自动选首个可用者）
  "stage": "writing",          # concept|world|characters|outline|scenes|writing
  "skillId": "wuxia",          # 插件技能 ID（决定写作风格指导）
  "message": "你的指令..."     # 具体任务描述
}
```

**响应**：
```json
{ "runId": "d2e0ed05-...", "conversationId": "conv_xxx" }
```

**事件流**：通过 SSE 实时获取 agent 事件：
```
GET /api/runs/{runId}/events      # SSE 流
```

### 3.3 质量检查

```bash
# AI 味检测（检测碎片化句型、重复词等）
POST /api/projects/{projectId}/check/ai-patterns
{ "chapterNum": 18 }

# 一致性检查（角色状态、时间线连贯性）
POST /api/projects/{projectId}/check/consistency

# 节奏检查
POST /api/projects/{projectId}/check/pacing
```

### 3.4 章节管理

```bash
# 列出所有章节
GET /api/projects/{projectId}/chapters

# 重新同步磁盘章节到 DB
POST /api/projects/{projectId}/chapters/resync

# 导出
POST /api/projects/{projectId}/export
```

## 4. 写作管道详解

### 4.1 一次写作 run 的完整生命周期

```
用户 POST /api/runs
    │
    ├─ 1. 创建 conversation + 存储用户消息
    ├─ 2. composePrompt() 组装分层上下文
    │      ├─ 核心设定层：concept.md + world-building.md（全文）
    │      ├─ 状态层：state.json（角色位置/情绪/已知信息/关系）
    │      ├─ 本章大纲块：outline-detailed.md 第N章片段
    │      ├─ 本章出场角色层：profiles/{角色}.md 分级注入 + 声口库（可选）
    │      ├─ 滚动摘要层：最近章节的 .summary.md
    │      └─ 伏笔层：foreshadow.json 中 status=pending 的伏笔
    │      ├─ 3. launchAgent() 启动 agent 子进程
    │      │      ├─ Claude Code：stream-json 模式（stdin 写 prompt）
    │      │      ├─ OpenCode：--prompt 单发模式（stdout 读流）
    │      │      └─ omp：ACP 协议模式（stdin/stdout 双向 JSON-RPC，经 acp-bridge 驱动）
    │      │      └─ agent 读大纲/场景/前文 → 写章节 → 写摘要 → 更新 state
    │      ├─ 4. stream-parser / acp-bridge 解析 agent 输出流（text/tool_use/tool_result）
    ├─ 5. child.on('close') 收尾：
    │      ├─ collectWrittenPaths：从事件提取写入的文件路径
    │      ├─ transformStreamEvents：合并 text/thinking delta → 持久化格式
    │      ├─ 存储assistant消息到 messages 表（events + artifacts 列）
    │      ├─ syncFilesToDb：文件变更同步到 DB
    │      ├─ ensureContextArtifacts：兜底补摘要 + 修复 state.json
    │      └─ createSnapshot：git commit 快照
    └─ 6. 返回 { runId, conversationId }
```

### 4.2 上下文分层（关键设计）

写作阶段的 prompt 由 `composePrompt()` 组装，分六层注入：

| 层 | 来源 | 作用 | 注入条件 |
|---|---|---|---|
| 核心设定 | concept.md + world-building.md | 恒定世界观锚点 | 写作阶段（writing/drafting/revision/polish） |
| 状态 | state.json | 角色当前位置/情绪/已知/关系 | 文件存在时 |
| **本章大纲** | **outline-detailed.md 第N章块** | **本章发生什么** | **写作阶段（大纲块提取）** |
| **本章出场角色** | **profiles/{角色}.md 分级注入** | **角色是谁、出身、驱动力、声口** | **写作阶段（三级回退识别）** |
| 滚动摘要 | 第N-K章的 .summary.md | 前文回顾（避免全量塞入） | buildRollingSummaryContext() |
| 活跃伏笔 | foreshadow.json | 待兑现的伏笔提醒 | status=pending 的条目 |

**出场角色分级注入**：POV 角色与显式出场角色按 L1 完整注入（出身/驱动力/性格等关键段，≤6KB/人）；仅被提及的角色按 L2 速查卡注入（~150 字）。角色层总预算 20KB，超出降级为 L2。声口库（`characters/voices/{角色}.md`）为可选资产，存在时附入 L1 角色。

**非写作阶段**（concept/world/characters/outline/scenes）不注入后三层，因为这些阶段还没有章节内容。

### 4.3 state.json 结构

```json
{
  "characters": [
    {
      "name": "林冲",
      "location": "翠微谷禁地",
      "emotion": "决心已定",
      "knows": ["令牌真相", "父亲守秘术"],
      "relationships": { "孙二娘": "脆弱的盟友" },
      "lastAppearance": 17
    }
  ],
  "timeline": "全书完",
  "activeForeshadows": [1, 3, 4],
  "foreshadowProgress": { "1": "承接（第15章...）" },
  "lastUpdatedChapter": 17,
  "updatedAt": "2026-07-03T17:30:00Z"
}
```

### 4.4 ensureContextArtifacts 兜底机制

agent 不总是可靠地写摘要和更新 state。`ensureContextArtifacts()` 在 run 结束时兜底：

1. **摘要兜底**：检测新写入的章节缺少 `.summary.md` 时，生成占位摘要
2. **state.json 修复**：`repairOrInitState()` 处理 LLM 常见 JSON 格式错误：
   - 键名含冒号：`"林冲:text"` → `"林冲": "text"`
   - 时间戳被拆：`"2026-07-03T18": "00:00Z"` → `"2026-07-03T18:00:00Z"`
3. **章节号更新**：从写入路径推断 lastUpdatedChapter

## 5. 批量写作

### 单章触发（推荐）

逐章触发，本章大纲与出场角色档案已自动注入：

```bash
curl -X POST http://localhost:3006/api/runs \
  -H "Content-Type: application/json" \
  -d '{
    "projectId": "proj_xxx",
    "agentId": "claude",
    "stage": "writing",
    "skillId": "wuxia",
    "message": "请写第N章正文。"
  }'
```

### 轮询等待完成

```bash
# 等待章节文件出现（agent 超时 30 分钟，建议轮询间隔 10 秒）
CHAPTER_FILE=".novel/chapters/第N章.md"
while [ ! -f "$CHAPTER_FILE" ]; do sleep 10; done
```

### 注意事项

- **不要并行写多章**：每章依赖前章摘要和 state，必须串行
- **监控退化**：长上下文运行可能导致 token 退化（重复词暴增）
- **定期检查 state.json**：确认 `lastUpdatedChapter` 跟上实际进度

## 6. 技能插件（Skill）

写作风格通过插件控制。插件位于 `plugins/` 目录：

```
plugins/
  wuxia/
    SKILL.md          # 写作风格指导（含"坏→好"示例、反 AI 味模式）
    templates/        # 阶段模板
    state.json        # 初始状态模板
```

SKILL.md 在 `composePrompt()` 中注入到 system prompt，指导 agent 的写作风格。

## 7. 已知限制

| 问题 | 影响 | 临时方案 |
|---|---|---|
| state.json 更新不可靠 | 后续章节可能读到过时状态 | 手动检查并修正 state.json |
| 长运行 token 退化 | 中段章节可能出现高频重复词 | 单章触发，避免连续大量写作 |
| 字数控制不精确 | 章节长度差异可达 2-4 倍 | 在 message 中明确字数约束 |
| wyw + babel TS 冲突 | 首页可能渲染报错 | 待修复（linaria 插件 TS 解析问题） |

## 8. 目录结构

```
open-novel/
├── src/
│   ├── agent/
│   │   ├── prompt-composer.ts    # 分层上下文组装
│   │   ├── context-manager.ts    # 摘要/state/伏笔管理 + 兜底
│   │   ├── launch.ts             # 启动 agent 子进程（claude/opencode/omp）
│   │   ├── acp-bridge.ts         # omp ACP 协议桥接（JSON-RPC ↔ StreamEvent）
│   │   ├── stream-parser.ts      # 解析 agent 输出流
│   │   ├── artifacts.ts          # 提取写入路径 + 同步 DB
│   │   ├── quality-checker.ts    # AI味/一致性/节奏检查
│   │   └── run.ts                # run 生命周期管理
│   ├── api/routes/
│   │   ├── projects.ts           # 项目 CRUD + 模板生成
│   │   ├── runs.ts               # agent 触发 + SSE 事件流
│   │   ├── chapters.ts           # 章节管理 + 磁盘同步
│   │   └── check.ts              # 质量检查端点
│   ├── web/                      # React 前端
│   └── db/                       # Drizzle schema + PGlite
├── plugins/                      # 类型插件（wuxia 等）
├── docs/                         # 文档
└── drizzle/                      # SQL 迁移
```
